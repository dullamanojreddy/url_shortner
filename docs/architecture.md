# Architecture

## Overview

The Distributed URL Shortener is a microservices system composed of six independent Node.js services, a reverse-proxy gateway (Nginx), a relational database (PostgreSQL), a cache cluster (Redis), and an async event pipeline (Kafka → Analytics Workers).

## Services

| Service | Port | Responsibility |
|---|---|---|
| auth-service | 3001 | Register, login, JWT issuance, API key management |
| url-service | 3002 | Create / update / delete / list short URLs |
| redirect-service | 3006 | Cache-first lookup and 302 redirect |
| analytics-service | 3003 | Kafka consumer, click storage, analytics query API |
| qr-service | 3004 | On-demand QR code generation (SVG + PNG) |
| admin-service | 3005 | User management, health dashboard, abuse detection |
| nginx (gateway) | 80 | Path-based routing, rate limiting, SSL termination |

## Request Flows

### URL Creation

```
Client → POST /api/v1/urls
       → Nginx (rate limit check)
       → url-service
           → Validate JWT
           → Generate Snowflake ID → Base62 encode → short code
           → INSERT into PostgreSQL
           → Write-through to Redis (TTL 24h)
           → Return { shortUrl }
```

### Redirect (cache hit — <50ms path)

```
Browser → GET /:shortCode
        → Nginx
        → redirect-service
            → Redis GET url:<shortCode>   ← HIT
            → Check expiry
            → Publish click event to Kafka (async, non-blocking)
            → Increment click_count in DB (async, non-blocking)
            → 302 Redirect
```

### Redirect (cache miss)

```
Browser → GET /:shortCode
        → redirect-service
            → Redis GET url:<shortCode>   ← MISS
            → PostgreSQL SELECT
            → Redis SET url:<shortCode> (warm cache)
            → 302 Redirect
```

### Click Analytics Pipeline

```
redirect-service
    → publishClickEvent({ shortCode, ip, userAgent, referer, timestamp })
    → Kafka topic: url.clicks

analytics-service (consumer)
    → Parse user-agent → extract browser, OS, device
    → INSERT into clicks table
```

## ID Generation — Snowflake + Base62

Short codes are generated by:

1. Creating a 64-bit Snowflake ID:
   - 41 bits: millisecond timestamp (since custom epoch)
   - 10 bits: machine ID
   - 12 bits: sequence (4096 IDs/ms per machine)
2. Base62-encoding the integer → 8-char alphanumeric code

This guarantees global uniqueness without coordination, supports ~4096 URLs/ms per instance, and produces URL-safe codes.

## Cache Strategy (Cache-Aside + Write-Through)

- **Write-through**: on URL creation, immediately populate Redis
- **Read-through**: redirect service checks Redis before DB
- **TTL**: 24 hours (keys refreshed on access)
- **Eviction**: `allkeys-lru` on Redis for memory safety
- **Invalidation**: on URL update/delete, `DEL url:<code>`

## Database Sharding Approach (future)

For millions of URLs, shard PostgreSQL on `short_code` consistent hash:
- Shard 0: codes starting with [0-9A-F...]
- Shard 1: codes starting with [G-Z...]
- Each shard has read replicas for SELECT-heavy redirect traffic
